# Conditions, Effects, and Augmentations

## Purpose and terminology

This subsystem represents reusable conditions, standalone effects, item
effects, and the concrete modifiers active on spawned characters. User-facing
copy generally says “Effect”; `Augmentation` remains the internal modifier
model because it is shared across state, schemas, actions, items, conditions,
and formula evaluation.

## Definitions, applications, and projections

The system has three layers.

### Authored definitions

- `ConditionPreset` is a named public or GM-only condition whose inline
  `augmentation_templates` describe its mechanics.
- `StandaloneEffectDefinition` is an independently authored modifier that can
  be applied or removed by an action.
- Each `Item` may own inline `augmentation_templates` activated through
  equipment.
- A manual `Augmentation` may exist as an authored concrete modifier.

### Runtime applications

- `ActiveCondition` records one applied condition on an instance, including
  its source, application time/role/version, and materialized augmentation IDs.
- `StandaloneEffectApplication` records each active standalone definition on
  an instance and its stack index.
- Top-level `Augmentation` records are the concrete modifiers used directly or
  materialized for condition/equipment sources.

### Derived projections

`direct_effect_projections` records stable base and effective values for paths
touched by projected direct effects. It is private bookkeeping regenerated by
the augmentation service and never authored or sent to clients.

Models live in
[`backend/state/models/augmentation.py`](../../backend/state/models/augmentation.py)
and [`backend/state/models/condition.py`](../../backend/state/models/condition.py).
Runtime behavior is centralized in
[`backend/features/augmentations/service.py`](../../backend/features/augmentations/service.py).
Condition definition/application services live in
[`backend/features/sheet_admin/conditions/`](../../backend/features/sheet_admin/conditions/),
and standalone definition CRUD lives in
[`backend/features/standalone_effects/`](../../backend/features/standalone_effects/).

## The augmentation model

An `Augmentation` contains:

- a source type and identifiers linking it to an item, action, condition,
  application, or other origin;
- a target root/path and sheet/instance scope;
- one effect type;
- active/applied and applied-target state;
- a lifecycle owner (`manual`, `equipment`, `condition`, or `action`);
- declarative lifecycle metadata.

The effect types are:

- `formula_modifier`: a direct numeric effect that changes a stored target
  while active;
- `evaluation_formula_modifier`: an evaluation-time numeric operation selected
  by tags, action/formula/step IDs, or same-source-item context, without
  changing stored state;
- `roll_mode_modifier`: an advantage/disadvantage modifier applied only during
  matching check evaluation.

Targets are selected from backend-provided augmentation target metadata.
Arbitrary state roots and paths are rejected; state-root targets are modeled
but not currently supported by runtime application.

## Equipment flow

The state-sync mutation hook scans every instance's equipped, in-stock
equippable items and derives the desired equipment-owned augmentations. It
rewrites sheet-scoped item templates to the owning instance, adds/removes
instance augmentation bridges, and reconciles the concrete collection.

Direct equipment effects participate in the projection system. Projection
recalculation starts from a stable base, combines every desired direct effect,
and writes only changed effective values. Unequipping or depleting an item
shrinks the desired set and restores the base value without double application.
Evaluation-time and roll-mode effects remain stored modifiers selected during
formula execution.

## Condition flow

Applying a condition creates `ActiveCondition` and materializes each inline
template as a condition-owned augmentation with an instance bridge. Direct
condition effects mutate the stored value immediately; evaluation-time and
roll-mode effects become active selectors.

Removing the application reverses reversible direct operations and removes the
bridges, augmentations, and active condition. A direct `set` effect cannot
derive the value that existed before application and therefore cannot be
generically reversed. Authoring and runtime flows must avoid relying on
automatic restoration for that case.

The production apply path is an action step and records the action source,
actor role, timestamp, and state version. DMs also have a dedicated active
condition removal route. Condition definitions themselves remain pure and do
not own runtime augmentation IDs.

## Standalone effect flow and stacking

Applying a standalone effect validates an instance target and creates a
`StandaloneEffectApplication`. Direct standalone effects are projected from the
definition rather than materialized as concrete top-level augmentations;
evaluation paths derive matching modifiers from active applications.

`unique` stacking permits one application per definition/instance and treats
reapplication as a no-op. `stack` permits multiple applications up to
`max_stacks`; each application participates independently, so cumulative math
falls out of normal projection/evaluation. Removal clears the complete stack
for that definition and instance.

There is no dedicated GM application inspector control for manually refreshing
or decrementing standalone lifecycles. Application/removal is currently driven
by authored action steps, while definition CRUD remains DM-owned.

## Lifecycle behavior

Lifecycle metadata supports `manual`, rounds, turns, until-rest,
until-source-removed, scene, remaining count, expiry timestamp, source-removal
intent, and notes. It is declarative. The backend has no turn/round/rest/scene
engine and does not tick, expire, or refresh these fields automatically.

The enforced lifecycle behavior is source-linked teardown for equipment and
conditions. Automatic turns, rests, and encounter timing would be a separate
future system and must not be implied by the current metadata.

## Redaction, deletion, and rebuild

Player state includes only visible applications for the assigned instance.
GM-only conditions and their materialized augmentations are removed, other
instances' applications are omitted, and direct projections are always
private. Definition/item redaction follows the same server-side visibility
rules as the rest of state sync.

Deleting an instance removes its conditions, standalone applications,
application-owned augmentations, and bridges. State import reconstructs the
model and re-runs equipment/direct-effect synchronization. Stable projection
bases make a valid round trip a no-op instead of double-applying modifiers.
